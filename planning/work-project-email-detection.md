# 工作项目讨论：邮件钓鱼检测系统

> 记录时间：2026-06-02  
> 目的：记录当前工作项目背景，以及后续如何将 AI / Agent 学习方向与真实业务项目结合。

---

## 一、项目情况简介

当前工作项目是一个邮件大批量实时检测服务，核心分为两个部分：

1. 邮件大批量聚类服务
2. 邮件实时检测服务

### 1. 邮件大批量聚类服务

该服务周期性处理邮件日志。

核心流程：

- 每 5 分钟从 ES 中搜索过去 5 分钟的邮件日志。
- 提取邮件文本特征。
- 文本特征由邮件的 **主题 + 附件名 + 域名** 拼接而成。
- 将文本特征向量化为 512 维数据。
- 结合本地缓存中已经处理好的历史数据，提取过去 1 小时的邮件日志。
- 对 1 小时窗口内的数据执行 DBSCAN 聚类。
- 对聚类结果进行邮件出入度、数量判断、白名单过滤等处理。
- 得到最终冒泡的聚类结果。
- 从聚类结果中提取三类基线特征：
  - md5
  - url_domain
  - 文本特征向量
- 将这些基线数据写入 Redis，供实时检测服务使用。

### 2. 邮件实时检测服务

实时检测服务是一个 FastAPI 服务。

核心流程：

- 服务定期从 Redis 更新基线。
- 提供检测接口。
- 输入为单封邮件数据。
- 输出为该邮件是否属于钓鱼邮件。

检测逻辑：

```text
白名单校验
→ md5 校验
→ url_domain 校验
→ 语义聚类检测
→ 输出检测结果
```

其中：

- 如果命中白名单，则认为正常。
- 如果命中 Redis 基线中的 md5，则判定为钓鱼邮件。
- 如果命中 Redis 基线中的 url_domain，则判定为钓鱼邮件。
- 如果前面步骤都未命中，则对邮件文本特征进行向量化，并与基线中的文本向量进行语义聚类检测。
- 如果语义聚类成功，则判定为钓鱼邮件。
- 其余情况判定为正常邮件。

### 3. 当前工作内容

当前已经完成或正在进行的工作包括：

- 两个核心服务的开发。
- 使用 AI 辅助接口性能调优。
- 对上下游数据源和依赖资源做监控和拨测。
- 对每日基线数据和检测命中的钓鱼数据做统计报告和数据分析。

技术栈：

- Python
- FastAPI
- ES
- Kafka
- S3
- Redis

## 二、对项目的看法和分析

这个项目不是普通的脚本或简单检测服务，而是一个具备较完整工程形态的安全检测系统。

它本质上包含三条链路：

```text
离线聚类基线生成
实时检测服务
检测链路观测与治理
```

从工程价值看，它具备以下特点：

- 大批量日志处理
- 准实时检测
- 向量化语义特征
- 聚类算法
- Redis 基线服务
- FastAPI 在线检测接口
- ES / Kafka / S3 / Redis 多组件协作
- 监控和拨测
- 检测结果统计
- 误拦风险治理

这些特征使它非常适合成为真实工作案例，而不仅是普通业务开发经历。

### 1. AI 不适合直接进入实时检测主链路

当前实时检测接口逻辑明确、确定性强，并且需要关注性能、稳定性和误拦风险。

因此，不建议在当前阶段将 LLM 直接放入实时检测链路中，也不建议让 AI 直接决定是否拦截邮件。

更合理的方式是：

```text
检测引擎负责高吞吐、低延迟、可确定的判断
AI 层负责复杂语义、风险分析、异常归因和人工辅助
```

### 2. 日报生成不是最核心的 AI 价值点

每日统计报告可以通过固定脚本完成：

- 约束好数据输入
- 约束好数据处理规则
- 约束好数据输出格式

LLM 可以在报告润色、自然语言总结方面提供帮助，但这不是最有价值的 Agent 落点。

更有价值的方向是让 AI 参与：

- 风险研判
- 异常归因
- 误拦治理
- 人工复核辅助

### 3. 真正的业务痛点是基线误拦风险

目前冒泡聚类会直接进入 Redis 基线。

已知误拦主要来源是：

> 正常域名误入 Redis 基线。

这意味着系统的核心风险不是“聚类是否能解释”，而是：

```text
这个冒泡聚类是否真的应该进入基线？
其中的 url_domain 是否可能是正常业务域名？
它进入 Redis 后是否可能造成误拦？
```

因此，最值得关注的是 **基线质量治理** 和 **正常域名误入风险发现**。

## 三、将 Agent 学习运用到该工程中的方向和建议

当前最适合的方向是：

> 邮件钓鱼检测系统中的 AI 辅助基线风险治理与可观测性增强。

这不是用 AI 替代检测规则，而是让 AI 在旁路辅助分析和风险治理中发挥作用。

### 1. 旁路 AI 基线风险审计

建议优先考虑一个旁路能力：

> AI 基线误拦风险审计器。

它不改变 Redis 基线写入逻辑，不阻塞实时检测接口，只对已经冒泡或即将进入基线的聚类做旁路风险分析。

整体位置：

```text
聚类服务
→ 冒泡聚类结果
→ Redis 基线
        ↓
  旁路 AI Baseline Auditor
        ↓
风险评分 / 告警 / 人工处理
```

它关注的问题包括：

- 聚类中的域名是否可能是正常业务域名。
- 域名历史通信频率是否较高。
- 该域名是否第一次出现。
- 该域名进入基线后是否可能造成误拦。
- 是否需要人工复核、移出基线或加入白名单。

这个方向适合以旁路方式验证，不影响主链路稳定性。

### 2. 检测链路可观测性和异常归因

该项目天然适合发展检测链路可观测性能力。

可以围绕以下对象建立诊断视角：

- ES 日志量
- Kafka 消费情况
- 聚类任务状态
- Redis 基线更新时间
- S3 写入结果
- FastAPI 检测接口延迟和 QPS
- 每日冒泡聚类数量
- 检测命中数量
- 白名单过滤比例
- md5 / url_domain / 语义聚类命中分布

未来可以发展为检测链路诊断 Agent：

```text
用户提出异常问题
→ Agent 查询相关监控和统计数据
→ 分析可能原因
→ 输出排查建议
```

例如：

```text
为什么今天 10 点后检测命中数突然下降？
```

Agent 可以围绕上游数据、聚类任务、Redis 基线和检测接口状态进行辅助归因。

### 3. 人工复核闭环

由于当前误报样本较少，适合先从轻量闭环开始：

```text
AI 旁路分析
→ 高风险误拦告警
→ 人工确认
→ 记录处理结果
→ 沉淀少量高质量样本
```

这个过程可以为后续更复杂的 Agent 工作流提供基础。

长期来看，可以逐步形成：

- 误报样本库
- 正常域名复核记录
- 白名单调整建议
- 基线准入风险记录
- 检测策略复盘材料

### 4. 与个人学习路线的关系

这个工作项目可以对应到当前学习路线中的多个能力：

- Prompt Engineering：用于风险分析、异常总结和人工复核建议。
- Structured Output：用于让 AI 输出固定格式的风险审计结果。
- Tool Calling：未来可让 Agent 查询域名历史、白名单、聚类样本和监控指标。
- RAG / Memory：未来可接入历史误报记录、检测策略文档、处理手册。
- Agent Workflow：未来可升级为基线风险诊断 Agent 或检测链路诊断 Agent。
- Human-in-the-loop：AI 只做旁路建议，最终由人工确认。

### 5. 推荐的阶段性方向

短期方向：

> 旁路 AI 基线风险审计，不影响实时检测逻辑。

中期方向：

> 基线风险告警 + 人工复核闭环。

长期方向：

> 检测链路诊断 Agent 和安全运营辅助 Agent。

该工作项目可以作为未来求职中的真实业务案例：

> 在邮件钓鱼检测系统中，围绕离线聚类基线生成和实时检测链路，探索 AI 辅助基线误拦风险审计、检测链路可观测性和异常归因能力，在不影响主检测链路稳定性的前提下，提升正常域名误入基线场景的发现和治理能力。
